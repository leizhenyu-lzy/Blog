# Tokenizer 分词器


流程
1. 文本 通过 Tokenizer 转为 词
2. 词 通过 Vocabulary 映射为 数字
3. 数字 转为 Embedding 向量


# BPE (Byte Pair Encoding)

介于 **字符级(Character-level) & 词级(Word-level)** 之间的 分词方法

**核心逻辑** : 找到 语料中 出现频率最高的 一对连续子单元(Bigram)，将它们 合并成一个新的单元，加入词表


流程
1. 初始化 (Initialization)
   1. 将训练语料中的每个单词拆分为字符序列
   2. 在 **单词末尾** 添加一个特殊的结束符号 `</w>`
2. 迭代合并 (Iterative Merging)
   1. 统计 连续对 的频率，合并这个 pair 加入词表(原本的 字符还存在)
   2. 过程会一直持续，直到达到预设的 词表大小(Vocabulary Size) 或 再也没有频繁的组合


优势
1. **解决 OOV(out-of-vocab) 问题** : 即使遇到没见过的单词，词表里保留了基础字符
   1. 使用 `UNK` 字符 替换 没有见过的 基础字符
   2. 如果 Vocab 拼接 基础字符表，则不需要 `UNK`
2. **压缩效率** : 常见的长词会被合并为一个完整的 Token，减少了序列长度，提高了计算效率

Tokenize (推理阶段)
1. 根据在 训练阶段 学到的 **合并规则(Merge Rules)**
   1. **优先级** : 严格按照训练时学到的 **合并顺序 (Merge Rank)** 进行
   2. **训练时** : 频率最高的 pair 最先被合并，因此它的 Rank 最靠前
   3. **推理时** : Tokenize 过程会遍历所有可能的 pair，**优先合并 Rank 最小 (即训练时 频率最高)** 的 pair
2. ==注意== : 并不是简单的 **长词优先**，而是 高频优先
3. 在实际效果上，高频词往往会被合并成较长的 Token，所以看起来像是在匹配长词

Training (训练阶段)
1. **准备语料** : 假设语料库中有单词 `("hug", 10), ("bug", 5), ("pun", 12), ("bun", 4)`
2. **拆分字符** : 所有单词拆成字符序列，并加结束符 `</w>`
   - `h u g </w>` : 10
   - `b u g </w>` : 5
   - `p u n </w>` : 12
   - `b u n </w>` : 4
3. **统计频次** : 统计所有相邻字符对 (Pair) 的频率
   - `g` + `</w>` : 10 + 5 = 15
   - `u` + `n` : 12 + 4 = 16  <-- **最高频** (与 `n`+`</w>` 并列)
   - ...
4. **合并 (Merge)** : 将最高频的 `u, n` 合并为新 Token `un`
   - 记录规则 **Rule #1: (u, n) -> un**
   - **词表 (Vocabulary)** : 新增 `un`，但 **保留** 原有的 `u` 和 `n` (为了处理其他单词)
   - **更新语料** : 只有符合条件的 pair 会被替换
     - `h u g </w>` : 10 (`u` **保留**，因为后面不是 `n`)
     - `b u g </w>` : 5
     - `p un </w>` : 12 (`u, n` **==被消耗/替换==** 成 `un`)
     - `b un </w>` : 4
5. **迭代** : 重复统计、合并，直到达到预设词表大小
   - 下一轮统计中，`g` + `</w>` (15次) 或 `un` + `</w>` (16次) 可能会成为最高频
   - 记录规则 **Rule #2: (un, </w>) -> un</w>**
   - **最终产物** : 一个 **有序的规则列表 (Ordered Merge List)**，一条条 append 下去
     - Rule #1: `(u, n) -> un`
     - Rule #2: `(un, </w>) -> un</w>`
     - Rule #3: `(u, g) -> ug`
     - ...
     - *这个顺序非常重要，推理时必须按此顺序（或优先级）匹配*

**文件结构** (Vocab & Merge List)
1. **通常是分开存储的** (例如 HuggingFace Tokenizers, GPT-2/RoBERTa)
   - `vocab.json` : **词表** (Token -> ID 的映射)
     - `{"u": 1, "n": 2, "un": 3, "g": 4, ...}`
   - `merges.txt` : **合并规则表** (Ranked Merge List)
     - 每一行代表一次合并，行号即优先级 (Rank)
       ```text
       u n
       un </w>
       u g
       ...
       ```
2. **也有合并存储的** (例如 SentencePiece / Llama `tokenizer.model`)
   - Google 的 SentencePiece 将词表和规则打包成一个二进制文件 (`.model` 或 `.proto`)
   - 但逻辑上，它依然包含这两部分信息：Token ID 和 Merge Priority (通常隐含在 ID 的顺序里，ID 越小优先级越高)

**词表扩充 (Vocabulary Growth)**
1. **初始状态** : 词表里 **只有** 基础字符 (Base Characters) 或 字节 (Bytes)
   - `["a", "b", "c", ..., "z", "</w>"]`
   - 此时，`a` 的 ID 可能是 0，`b` 是 1，`z` 是 25...
2. **训练过程** : 每执行一次 Merge (例如 `u, n -> un`)，产生的新 Token `un` 就会被 **加入** 词表
   - 新 Token `un` 的 ID 通常是 **顺延** 的 (例如基础字符占了 0-255，那么 `un` 的 ID 就是 256)
3. **最终状态** :
   - **Token ID = 它在词表列表中的索引 (Index)**
   - **Vocabulary = 基础字符 + 所有 Merge 产生的 Token**
   - 词表大小 = 基础字符数 + Merge 次数
   - *例如：GPT-2 的词表大小是 50257 (256 个基础字节 + 50000 次 Merge + 1 个特殊符号)*

**语料库的影响 (Impact of Corpus)**
1. **不同语料 -> 不同规则** :
   - 虽然算法逻辑永远是 **"频率优先"**，但不同语料的 **词频分布** 完全不同，导致学到的规则和顺序也不同
2. **例子** :
   1. **英文小说语料** : `th`, `he`, `ing` 频率极高，Rank 靠前
   2. **Python 代码语料** : `im` (import), `de` (def), `    ` (4空格缩进) 频率极高，Rank 靠前
3. **通用性问题**
   1. 如果用 **英文** 训练的分词器去切 **中文**，或者用 **文学** 分词器去切 **代码**，效率会很低
   2. 因为分词器没学到那个领域的 常用套路，只能把词切得很碎（退化成字符级）



Detokenize
1. BPE 在训练时给单词末尾加了特殊的标记 `</w>`
2. 解码器 可以 根据标记 恢复空格

# BBPE (Byte-Level BPE)

基础词表不再是 Unicode 字符，而是 256($=2^8$) 个 基础字节 (Bytes)

无论什么语言、什么特殊符号(甚至表情符号)，都可以用这 256 个字节表示

保证了词表极其紧凑且能覆盖全球所有文本

传统 BPE 是基于 字符(Character) 的，**Unicode 字符** 有 14 万个，不可能把它们全放进词表(通常词表只有 3~5 万)，会有漏网之鱼(变成 UNK)

BBPE 是基于 字节 (Byte) 的，在 **UTF-8 编码** 下，只要词表里包含了 0x00 到 0xFF 这 256 个基础字节，就能拼出世界上 任何 一个字符、任何一个 Emoji、任何一个文件

**多字节字符的处理**
1. **UTF-8 特性** : 中文通常占 3 个字节，Emoji 通常占 4 个字节
2. **合并逻辑** : 虽然基础单元是单字节，但只要某个多字节组合 在语料中 **频率够高**，BPE 算法就会把字节逐步合并
3. **结论** : 高频的汉字或 Emoji 依然会被合并成 **一个独立的 Token**，只有生僻字才会被拆散成多个字节 Token
4. **解码 (Decode)** : 被拆散的 Token 怎么还原？
   - **拼接** : 解码器会将所有 Token 对应的 **字节序列 (Bytes)** 直接拼在一起
   - **UTF-8 解码** : 拼完后，得到一串完整的字节流 `0xE4 0xB8 0xAD ...`，再统一用 UTF-8 格式转回字符串
   - *只要字节顺序没乱，拆成 单字节 也能拼回原来的字*
